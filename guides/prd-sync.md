# PRD 동기화 가이드

## 문제: PRD 드리프트

```
PRD 작성 → 개발 시작 → 스펙 변경 발생 → PRD 안 고침
→ PRD와 실제 코드가 점점 달라짐 → 기획 리뷰가 무의미해짐
```

## 해결 규칙

### 1. feat: PR에 PRD 변경 포함 여부 체크

Claude Code가 리뷰 종합 시 아래를 확인합니다:

```
- 이 PR이 기존 스펙을 변경하는가?
  → Yes: ".project/PRD.md 업데이트가 필요합니다" 코멘트
  → No: 패스
```

### 2. PRD 업데이트 기준

| 상황 | PRD 업데이트 필요? |
|------|:------------------:|
| PRD에 있는 기능을 그대로 구현 | ❌ |
| PRD에 있는 기능을 다르게 구현 | ✅ 차이점 반영 |
| PRD에 없는 새 기능 추가 | ✅ 기능 추가 |
| 기존 기능 삭제/변경 | ✅ 변경 사항 반영 |
| 리팩토링 (동작 변경 없음) | ❌ |
| 버그 수정 (스펙 변경 없음) | ❌ |

### 3. PRD 업데이트 방법

**권장: 같은 PR에서 코드 + PRD 함께 수정**

```bash
# 기능 코드 수정
git add src/...

# PRD도 함께 수정
git add .project/PRD.md

# 하나의 커밋 또는 별도 커밋 모두 가능
git commit -m "feat: 종목 수 52개로 확장 + PRD 업데이트"
```

**차선: 별도 docs: 커밋으로 후속 처리**

```bash
git commit -m "docs: PRD 업데이트 — 종목 수 10 → 52개 변경 반영"
```

### 4. PRD에 넣으면 안 되는 것

- 구현 세부사항 (어떤 라이브러리를 쓰는지 등) → tech-spec.md에
- API 엔드포인트 목록 → api-spec.md에
- 디자인 상세 (색상, 간격 등) → design-spec.md에

PRD는 **"무엇을 만드는가"**에 집중합니다. "어떻게 만드는가"는 다른 문서에.

### 5. 주기적 PRD 감사

스프린트 회고 시 아래를 체크합니다:

```
- PRD에 적힌 기능 목록 vs 실제 구현된 기능 비교
- 불일치 항목 식별
- PRD 일괄 업데이트 또는 미구현 기능 백로그 이동
```

이건 자동화하기 어렵고, 스프린트 단위로 수동 점검이 현실적입니다.
